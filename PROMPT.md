# 一条 Prompt 制作或接管《沫沫的珊瑚午餐》

将下面代码块中的内容作为**一条完整 Prompt** 发送给具备代码、图片生成和本地文件操作能力的 Agent。它既可以从空目录完成游戏，也可以接管已有项目继续检查和修复，不需要用户补充需求或逐步确认。

如果只想实现或重做某一部分，可改用 [`README.md`](README.md) 中的模块 Prompt；每份都可以独立复制执行。

```text
你是一名同时具备 AI 生图、图片切割与透明背景处理、游戏设计、前端开发、音频整合、视觉测试和技术文档能力的制作代理。

请在当前工作区制作并交付一款名为《沫沫的珊瑚午餐》的完整网页小游戏。如果工作区为空，就从零制作；如果已有项目，就先检查现状并接管尚未完成或不稳定的部分。不要只给方案、审计报告、伪代码、教程或待办清单，也不要在发现普通问题后等待用户逐项指导；请自主做合理决定，直接修改、运行、检查、修复和回归，直到得到稳定可运行的独立资源版与单 HTML 版。

【最终目标】

制作一款可爱、完整、有明确成长反馈、儿童可以快速理解的横版海底“大鱼吃小鱼”游戏，而不是棋盘式探索或复杂三栏管理界面。玩家控制戴铜制探照灯、背蓝色小背包的橙白小丑鱼“沫沫”，在三片海域中追逐不同品种的小鱼和虾、躲避或反击刺豚与水母，并在每关结尾挑战鲨鱼王。游戏必须同时提供手动游玩和“水族馆观赏”自动游玩模式。

交付物必须能在普通静态服务器和任意子路径下运行，页面运行时不得请求外网。所有图片、视频、音乐和音效必须保存在项目内。以独立资源版为主要入口，并额外生成一个把所有资源内嵌进去的单 HTML 版本；不要求执行 GitHub Pages 发布或其他远程部署。

【工作方式与自动循环】

请完整执行生产流程，不要停留在规划阶段：

确认游戏概念和统一海底像素美术方向
→ 盘点角色、海洋生物、场景、景深、物品、地图、UI 和音频
→ 生成或补齐素材
→ 切图、透明化与锚点对齐
→ 检查素材
→ 开发或修复游戏
→ 在真实浏览器运行
→ 检查视觉、交互、逻辑和响应式
→ 修复问题
→ 重新测试

素材合集只能作为生图和切图的中间产物，不能直接把整张合集放进游戏。角色、海洋生物、场景、景深物件和 UI 图标必须使用切分后的独立透明素材。

如果素材存在裁切不完整、背景未真正透明、白边或品红残留、多个物体粘连、鱼鳍/鱼尾/触须/触手截断、比例异常、像素模糊、视角、光向或风格不一致、放入游戏后不协调、图片加载失败等问题，不要只报告；应根据根因自动选择重新生图、重新切片、重新抠图、连通域清理、最近邻缩放、重新对齐、调整素材或重新嵌入，然后再次检查。

如果游戏存在布局溢出或重叠、图片撑大容器、海底元素错位、视差、画布或路线缩放异常、文字难以阅读、中英数字字重不协调、交互无法使用、玩法流程不完整、自动模式卡死、页面缩放或响应式变形、媒体加载失败或脚本错误，也要直接修复并重新测试。

优先保证：海底世界丰富且有探索氛围；美术统一；前景清晰；透明素材真实独立；等距偏侧视一致；核心玩法从开始到三关结算完整；观赏模式可自动通关并可接管；桌面和移动端可用；音频与所有素材本地化；构建可复现；README、Prompt 文档、验证 JSON 和四组 contact sheet 与实际产物一致。

除非遇到确实无法继续的外部限制，否则持续执行生成、检查、修复和测试循环。每次修复都回归相关路径，不要把问题列表当作最终交付。

【技术与目录】

1. 使用原生 HTML、CSS、Canvas 2D 和 JavaScript，不依赖前端框架，不要求联网安装运行时依赖。
2. 游戏逻辑以 1280×720 横屏坐标系设计，但必须响应式适配桌面、平板和手机横竖屏；处理 safe-area、触控、窗口缩放和高 DPR。
3. 交付根目录至少包含：
   - index.html：独立资源版本入口。
   - assets/images/：运行时图片。
   - assets/audio/：BGM、音效和 CREDITS.md。
   - assets/media/：本地水面焦散视频；视频不可用时必须有 CSS/Canvas 后备效果。
   - workflow/sources/：生成原图。
   - workflow/tools/：素材透明化、切片、锚点对齐、验证、contact sheet 和构建脚本。
   - workflow/reports/：验证 JSON，以及动画、场景、景深、UI 四组 contact sheet。
   - README.md：运行、玩法、设计、素材流程、检查结果、音频许可和四张 contact sheet。
   - prompts/00-one-shot.md：保存本条完整制作与接管 Prompt。
4. 在项目同级或 dist/ 中额外生成一个全内嵌的 momo-reef-feast.html。内嵌版与独立资源版的玩法和视觉必须一致。
5. JavaScript 不得出现语法错误；Canvas 绘制启用 imageSmoothingEnabled = false。玩家位置保持浮点精度，只在绘制时像素对齐，避免移动抖动。

【核心玩法】

1. 输入：
   - 鼠标或触控移动目标，按住鼠标或手指冲刺。
   - 方向键或 WASD 移动，按住 Shift 冲刺。
   - 首次有效交互后启动音频。
   - 提供暂停、总声音开关、音量滑杆和观赏模式按钮。
2. 玩家从较小体型开始。吃掉比自己小的猎物会得分并增长；必须按画面中的实际视觉尺寸判断碰撞胜负，不能出现“看起来更大却仍受伤”。
3. 小鱼需要感知玩家距离、接近方向和速度并自然转向闪躲，不能只是直线机械飘过。不同物种不能只是同一条灯笼鱼换色，必须有不同轮廓、体型、速度、分值、游动节奏与反应方式。
4. 冲刺要有明确反馈：速度提升、身体轻微前倾、连续泡泡、水流尾迹和适度镜头/音效反馈；避免僵硬瞬移。冲刺状态提示使用紧凑的鳍、泡泡或圆形反馈，不要做成突兀的长矩形条。
5. 玩家有三颗泡泡心。碰到当前无法吞食的刺豚、水母或鲨鱼会损失一颗心，并触发短暂无敌、红色边缘闪光、HUD 弹性反馈、粒子和受伤音效。
6. 成长到 100% 后进入“无敌大胃王”阶段，可吞食刺豚和水母，并能攻击鲨鱼；成长值仍可继续到 180%。达到 100% 不得立即结束关卡。
7. 每关必须出现一只鲨鱼王。玩家在 100% 后连续成功咬中鲨鱼三次才能击败它；击败后显示明确反馈，并由玩家主动点击按钮进入下一海域。
8. 完成三片海域后显示总结与总分；泡泡心耗尽时显示失败页，可直接重新开始。
9. 吃普通猎物、吃大型敌人、泡泡爆裂、玩家受伤、升级、鲨鱼出现和通关都要有彼此可区分的动画、粒子、屏幕反馈和声音。每个物种还要有自己的小动作或环境反馈，例如灯笼鱼诱饵呼吸发光、水母扩散柔和波纹、刺豚鼓起、虾的触须和尾扇节奏、鲨鱼有重量的摆尾，不能只统一播放同一种位移动画。

【水族馆观赏模式】

1. 开始页提供“水族馆观赏”入口，游戏 HUD 也提供珍珠按钮开关。
2. 自动模式必须真正完成游戏：自主寻找合适猎物、转向、冲刺、避开当前危险、成长后反击、挑战鲨鱼、进入下一关，并在三关结束后循环巡游。
3. 自动游玩要像自然的小鱼行为，不要逐帧锁死目标；加入目标评估、危险预测、转向阻尼和少量游动节奏变化。
4. 观赏时点击/触摸画面或按方向键/WASD 可立即由用户接管；再次点击 HUD 观赏按钮可恢复自动游玩。模式变化要更新提示文字和 aria 状态。

【三片海域】

请实现三个数据驱动关卡，每关都改变色调、流速、猎物密度、危险组合、背景地貌和鲨鱼节奏：

1. 珊瑚暖湾：明亮钴蓝与珊瑚暖色；水流较慢；蝴蝶鱼、小虾、灯笼鱼较多；刺豚和水母较少。用于教学和建立食物链认知。
2. 幽蓝沉船道：更深的蓝色；水流加快；加入沉船、石柱和成对水母；猎物更灵活，危险生成更频繁。
3. 月光海沟：靛蓝与深海紫；水流最快；尖塔、石拱与群怪组合更密集；鲨鱼更有压迫感，但仍保持可读和公平。

HUD 的关卡文字要显示“当前关卡/3、海域名、当前体型阶段”。移动端可缩短文字，但不得裁切或互相覆盖。

【角色与动画素材】

整体美术 DNA：可爱、明亮轮廓、深海蓝环境、等距偏侧视、手绘像素颗粒、深海军蓝描边、暖色可交互主体、清晰的最近邻像素边缘。角色统一朝右的三分之四侧视角。

必须制作以下 8 组角色动画：沫沫、灯笼鱼、蝴蝶鱼、沙丁鱼、小虾、刺豚、水母、鲨鱼王。每组先生成一张严格 3×2、六个等尺寸格子的 sprite sheet 原图，再自动切成六帧、透明化、对齐锚点，最后合成横向运行时 spritesheet。

生成角色原图时统一使用这一视觉要求：cohesive handcrafted isometric pixel-art game sprite sheet, cute underwater adventure, crisp nearest-neighbor pixels, dark navy outline, warm readable highlights, consistent three-quarter side view facing right, exactly six equal cells in a 3 by 2 grid, one complete isolated subject per cell, every fin tail antenna and tentacle fully visible, no overlap between cells, no text, no shadow, solid flat chroma-magenta background #ff00ff。

角色差异：
- 沫沫：橙白小丑鱼、圆润 Q 版比例、大眼睛、铜制头灯、蓝色小背包；六帧轻微尾摆和身体起伏；头灯与背包外形在所有帧固定。
- 灯笼鱼：金色、圆润、额前暖黄诱饵；尾摆时灯泡轻微呼吸，但诱饵根部稳定。
- 蝴蝶鱼：黄黑白条纹、扁平菱形身体、长背鳍和腹鳍；动作优雅。
- 沙丁鱼：银蓝细长、快速小幅尾摆，适合群游。
- 小虾：珊瑚粉、完整长触须、分节小脚和尾扇；不得截断触须。
- 刺豚：紫色、圆润、可爱但警觉；六帧围绕固定中心从收缩到鼓起再恢复。
- 水母：柔粉半透明钟体、可爱表情、完整卷曲触手；钟体顶部固定，触手不得贴边。
- 鲨鱼王：蓝灰色、宽头、明显牙齿和背鳍，可爱但有重量；六帧缓慢有力的尾摆。

动画验收：总计 8 张运行时 spritesheet、48 帧；透明四角正常；无品红/紫色残留；主体、鱼鳍、触须和触手不触边；同一角色的视角、比例和光向一致；最大锚点误差不超过 0.5 px。重点在移动中而不只在静止截图检查抖动：每帧画布、裁切范围、视觉中心、锚点和绘制尺寸稳定，玩家坐标只在最终绘制时统一像素对齐，不能因逐帧透明边界或重复取整产生跳动。

不要把所有物种机械输出为同一分辨率。先统计每个角色在游戏中的最大绘制尺寸、成长倍率和目标 DPR，再反推源图与单帧像素预算；鲨鱼王等大型主体必须使用更大的源图/帧或高分辨率变体，不能把低分辨率 256 px 帧持续放大。最近邻缩放只能保持像素边缘，不能凭空补充细节；在最大体型和高 DPR 浏览器截图中仍要轮廓清楚、细节可读，不能模糊或粗糙失真。

如果当前环境具备图片生成能力，直接生成素材；如果没有，就用 Canvas/Pillow 等方式程序化绘制有质量的原创像素素材。不得因为缺少生图工具而停止，也不得用外链图片占位。

若未来环境提供可靠的视频生成能力，优先评估“短循环角色视频 → 定距抽帧 → 去背 → 像素化 → 锚点对齐 → spritesheet”的动画路线。视频生成通常比让 ImageGen 独立生成六帧更有时间连续性，能表现更灵活、复杂的摆尾、转向、鼓起和触手运动。本次环境若没有视频生成工具，就继续使用 3×2 ImageGen sprite sheet，不得伪造已使用视频。视频路线也必须固定镜头、构图、朝向、循环首尾和纯色背景，并检查形体漂移、身份变化、运动模糊与镜头移动；只有抽帧后通过同一验证流程才能接入游戏。

【场景、景深与水下光影】

1. 制作透明场景物件：左右海床转角、气泡、珊瑚、海草、海葵、残破石柱、张开的珍珠贝和沉船。底部接地自然，组合时不能露出明显矩形边缘。可以分析工作区中用户提供或已有的参考项目、截图、视频和素材，提取构图、景深与水下氛围规律；除非许可与风格都合适，否则不要直接照搬，需重新绘制或转换成统一像素风。
2. 制作景深素材：远景海床、远景山脊、中景海床、岩壁、岩架、石拱和尖塔。
3. 画面至少使用三层视差：低对比远景海床、中景石拱/尖塔、可进入画面边缘遮挡的前景岩壁/岩架。不同层的移动速度、饱和度、亮度和细节量要有区别。
4. 叠加本地水面焦散视频，可使用 screen 或适合的混合模式；同时制作 CSS 或 Canvas 静态/缓动焦散后备层。不能只确认文件存在，要在浏览器读取 readyState、实际播放状态、计算透明度和画面可见性；焦散应清楚可见但不能压过角色识别度，可从约 0.2 的透明度附近按实测调整。
5. 玩家头灯形成范围足够的柔和锥形聚光灯。光束起点必须和头灯锚点共用同一个旋转/翻转矩阵，因此上下游动、左右翻转、键盘控制和缩放时都不能错位；指针靠近玩家时沿最近稳定方向保持最小探索长度，不能塌缩或消失。键盘输入只更新稳定朝向，不应让光束长度突然跳变；边缘用多层渐变或遮罩羽化，避免硬三角形。
6. 聚光灯外的区域应比光照内更暗、对比更低或略去饱和，形成明确探索氛围。前景角色仍保留暖色辨识度，再通过轻度青蓝 tint、深度亮度和聚光灯统一进海水环境，不能像贴纸浮在背景上。第一关也要保留较暗的深海层次，不能亮到平淡或洗白。
7. 在所有目标尺寸检查背景覆盖、海床拼接和视差边界，不能在底部或侧边出现黑块、断层、矩形接缝或因 cover/缩放产生的空洞。

【UI 与文字】

1. UI 采用已经验证稳定的 Glass + Water 混合方案：圆润、半透明的 CSS 玻璃容器负责动态文字、宽度、换行和响应式布局；统一像素风的独立透明图标、角饰和少量固定装饰负责游戏感。不要把整张框体直接拉伸，也不要让每块 HUD 都变成厚重边框。只有不承载复杂动态尺寸、且经过多断点验证的主对话框可以使用九宫格或 `border-image`。
2. HUD 至少包括：得分、成长条与百分比、体型/海域、三颗泡泡心、暂停、观赏模式、音乐开关和音量。
3. 开始/结束对话框、toast、底部操作提示、下一关按钮都必须响应内容宽度和屏幕尺寸。CSS flex/grid 负责内容布局；文字不能烘焙进图片。若某个固定装饰框使用九宫格或 `border-image`，必须明确 inset、固定边角和安全中心区，并在所有目标断点验证。出现拉伸、黑边、错位或 padding 失衡时，优先缩小切图职责，让动态主体回到 Glass + Water 容器，而不是继续叠加覆盖规则。
4. 制作 4×4 UI atlas，共 16 个基础素材：HUD panel、growth bar frame、lives panel、round button、dialog frame、toast panel、tip panel、action button、star、heart、pause、music、play、boost、pearl、warning。另生成 1 个独立成长图标，造型使用金色扇贝、青色上升箭头与少量气泡，不能和底部冲刺鳍复用同一语义。总计验证 17 个独立透明 UI 素材；每格/每图一个素材，无文字、无重叠、品红色键背景。
5. Glass + Water 容器采用半透明青蓝渐变、胶囊轮廓、内高光和克制阴影；不要依赖粗边框制造层级。`backdrop-filter` 仅作增强且必须有清晰可读的后备。背景限制在容器内部，像素装饰外不得出现矩形黑底、额外大阴影或被拉伸的边角。
6. 全部可见文案使用简体中文，只有封面副标题可使用英文 “A TINY FISH, A BIG APPETITE”。
7. 字体优先使用 PingFang SC、Hiragino Sans GB、Microsoft YaHei、Noto Sans CJK SC 和 system-ui，禁止远程字体。中英文与数字要视觉协调：不要把 Avenir 等偏黑的西文字体放在中文字体之前；正文、HUD 数字、百分比和封面英文使用约 600–700 权重，不要用 900/950 造成英文和数字明显比中文更粗。数字使用 tabular-nums。
8. 所有按钮具有 hover、active、focus-visible 和触控反馈；主按钮和图标按钮保持圆润，hover 只能产生轻微亮度、浮起或水光变化，不能变形、切角、错位或突然出现错误底色。可点击区域至少 44×44 px。提供语义化 aria-label、aria-live、role=meter 和 aria-pressed。
9. 图标必须语义明确并完成资源映射；成长图标放在标签/进度条左侧等自然位置，不悬空压在底部。相邻功能不要重复使用同一图标，例如成长与冲刺提示应能一眼区分。检查图标是否实际渲染、成长条/音量等渐变填充是否可见、文字与图标是否垂直居中、上下 padding 是否平衡、核心面板是否有尴尬空洞。

【音频】

1. 使用可再分发的本地 CC0 或等价许可的真实录音/设计音频，并在 assets/audio/CREDITS.md 写明名称、来源链接、作者、修改和许可。可以联网寻找并下载合适素材，但运行时必须完全本地化；不要用单调振荡器合成音代替主要 BGM、吃鱼或受伤音效。
```
